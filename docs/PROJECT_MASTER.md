## 1. Operating Rules

1. `docs/PROJECT_MASTER.md` is the only project status document.
2. Do not create new `*_REPORT.md`, `*_SUMMARY.md`, `*_COMPLETE.md`, or similar status files.
3. Dedicated setup documents are allowed only when they are operational references, such as OAuth setup or deployment guides.
4. Every meaningful change must update the Progress Ledger in this file.
5. GitHub sync target for this rewrite is `https://github.com/bedlatess/PDF_Flow_v2`. Keep the old `origin` repository available as legacy reference; sync to the v2 repository after each major category, not after every small task.
6. Cleanup must be deliberate:
   - Generated files can be removed after their paths are verified.
   - Source files, migrations, tests, and deployment scripts require review before deletion.
   - User or historical requirement material should be preserved unless explicitly retired.
7. Verification evidence matters. A change is not complete until the relevant build, tests, or manual checks are recorded.

## 2. Product Direction

PDF-Flow will be rebuilt around a workspace-first product experience.

The chosen direction is:

- Primary style: a calm, professional PDF workspace.
- Tool access: dense and efficient like iLovePDF, especially for homepage tool discovery and the tools center.
- Advanced capability expression: AI, enterprise, admin, diagnostics, and automation should feel powerful without overwhelming first-time users.

Reference influence:

- iLovePDF: fast tool grid, clear categories, direct conversion paths.
- Smallpdf: clean trust-building flow, single-tool focus, polished SaaS feel.
- LightPDF: AI positioning, broader platform narrative, document intelligence angle.

PDF-Flow should not copy these sites. It should use their lessons while keeping its own differentiator: local-first processing with optional cloud enhancement.

## 6A. Target Payment Architecture

Supported providers:

- Stripe
- PayPal
- 易支付
- 支付宝
- 微信支付
- TokenPay
- BEPUSDT
- EPUSDT
- OKPay

Payment principles:

- Frontend never decides payment success or account entitlement.
- Backend creates every payment order with a merchant order id, amount, currency, user id, provider, expiry, and status.
- Provider adapters own signing, request shape, callback parsing, and webhook verification.
- Provider notification URLs are generated from backend `BACKEND_PUBLIC_URL` and must point to `/api/v1/payment/webhooks/{provider}`; frontend success/cancel URLs are only return-page UX.
- The admin payment view surfaces each provider's exact webhook/notify URL, frontend success/cancel return URLs, merchant-console hint, required backend config keys, and missing config keys so merchant dashboards can be configured without reading code.
- The admin payment view also carries a provider-specific payment integration runbook: sandbox smoke steps, go-live checklist, expected event flow, troubleshooting actions, and evidence fields to capture during manual provider testing.
- The admin payment view can copy a safe payment integration evidence packet that combines deployment URLs, manual test placeholders, latest order/event state, expected flow, troubleshooting steps, and evidence fields while excluding checkout URLs, raw provider payloads, document contents, and secrets.
- The admin payment view shows a provider acceptance matrix (`disabled`, `missing_config`, `ready_to_test`, `waiting_callback`, `needs_review`, `accepted`) so each payment method has a clear next operational action. This matrix reflects local evidence only; final production acceptance still requires merchant-dashboard callback configuration and sandbox/live smoke tests.
- Webhooks/callbacks normalize into one internal payment event format.
- Entitlement granting is provider-neutral and happens only after amount/currency/order validation.
- One-time providers extend a local entitlement period; true subscription providers can later add provider-side recurring cancellation/reactivation.
- Admin exposes first-pass provider health, recent orders, safe reconciliation packets, and recent payment event processing state. Manual reconciliation controls remain a later operations feature.
- Payment event records are append-style operational ledgers keyed by provider event id. Replayed provider webhooks or repeated capture attempts must not grant entitlement twice.

Initial provider strategy:

1. Stripe remains the default compatibility provider for the current pricing checkout route.
2. PayPal has a real first-pass adapter: OAuth2 access token, Orders v2 checkout creation, capture endpoint, and PayPal webhook signature verification.
3. Alipay has a real first-pass adapter for hosted checkout creation, RSA2 request signing, and signed notification verification.
4. WeChat Pay has a real first-pass adapter for Native prepay order creation, RSA request signing, platform signature verification, and API v3 resource decryption.
5. EPay, TokenPay, BEPUSDT, EPUSDT, and OKPay share configurable hosted-gateway adapters with signed checkout URLs and strict callback signature verification.
6. No provider is exposed in production unless credentials, signing, callback URL, and reconciliation behavior are tested against its sandbox or live merchant account.

## 7. Target Visual System

Product tone:

- Calm, practical, trustworthy.
- More workspace than marketing site.
- Polished but not decorative.
- Dense enough for repeated tool use.

Visual principles:

- Use clean sections, not nested decorative cards.
- Use cards for repeated items, modals, and actual framed tools.
- Keep tool cards compact and scannable.
- Use icons for tool identity and controls.
- Avoid one-note purple/blue gradients.
- Favor a neutral workspace base with meaningful accent colors by tool category.
- Text must fit on mobile and desktop.
- Component dimensions should be stable to avoid layout shifts.

Suggested token direction:

- Surface: white, off-white, near-black for admin.
- Text: slate/neutral scale.
- Primary accent: PDF red or controlled coral for main actions.
- Secondary accents: teal, amber, blue, emerald by category.
- Border radius: mostly 8px; larger only for hero/workspace frames when justified.

## 9. Backend and Data Flow Rules

Processing model:

- Prefer local browser processing for core daily tasks when feasible.
- Use cloud processing for OCR, Office conversion, AI, very large files, or operations that need backend libraries.
- Keep feature flags as the public contract for availability, login requirements, Pro requirements, and maintenance messages.

Security and privacy:

- Do not log file contents.
- Do not expose internal stack traces in production UI.
- Keep upload retention and cleanup explicit.
- Preserve audit logs for admin actions.
- Keep API key and JWT handling server-controlled.

Admin model:

- Control Room is an operations console, not a marketing dashboard.
- Every destructive action needs confirmation.
- Bulk cleanup must show counts and scope before execution.
- Diagnostics should provide enough context for support without leaking user document contents.

## 11.1 Phase Governance And Launch Gates

Current operating stage:

- P0 is closed. Reopen P0 only for blocking runtime failures, security/privacy risks, auth/checkout/payment breakage, data-loss risks, or user-facing flows that fail without a visible recovery path.
- P1 is functionally closed. Reopen P1 only for public/account/admin UX debt that blocks normal use.
- Active work is P2/P3 hardening: payment operations readiness, backend residual cleanup, admin diagnostics, cloud/tool workflow coverage, performance guardrails, and launch-test preparation.
- P4/P5 work should wait until the current P2/P3 launch-test readiness pass is stable, unless a hard architectural blocker appears.

Recommended order from here:

1. Keep closing P2/P3 residual risks in small verified batches.
2. When local gates are green, deploy to `staging` and run the full smoke suite.
3. If `staging` passes, merge to `main`, run `deploy-main.sh`, and run `main-smoke-suite.sh`.
4. Treat real merchant payment acceptance as a separate operations gate: credentials, callback URLs, sandbox/live low-value tests, and provider dashboard evidence are required before claiming payment production acceptance.
5. Start the independent backend/domain-service split only after the first server launch-test pass, so deployment, CORS, env, and API-contract churn do not destabilize the first real validation.

New GitHub repository rollout path:

- Preferred strategy: keep the old GitHub repository and old server deployment untouched while this rewrite is uploaded to a separate new repository.
- Local publication helper: `scripts/prepare-new-repo-push.sh`.
- Server replacement helper: `scripts/server-clean-reclone.sh`.
- Both helpers default to dry-run mode. They require explicit repository and target path environment variables before doing anything real.
- The local helper pushes the current committed `HEAD` to a new remote name such as `new-origin` and refuses to overwrite `origin` unless explicitly allowed.
- The server helper creates a timestamped backup folder, runs the optional backup command before stopping services, stops old Docker Compose services when present, moves the old target directory into `old-project`, clones the new repository, restores `.env`/compose override files from `old-project`, then runs the existing deploy and smoke scripts.
- The old server directory should not be physically deleted until the new repository clone passes server smoke checks and manual browser checks.
- This path replaces the earlier idea of deleting the old GitHub repository contents in place; a separate repository gives a cleaner launch channel and keeps the old production code available as an emergency reference.

Launch-test readiness gate:

- Required local gates: backend pytest for touched domains, frontend unit tests, type-check, build, and `test:e2e:core`.
- Required server gates: `smoke-test.sh`, `business-smoke-test.sh`, `ocr-smoke-test.sh`, and `office-smoke-test.sh`; on `main`, prefer `scripts/main-smoke-suite.sh`.
- Required manual gates: homepage/tools, auth/account, pricing checkout provider selection, admin Control Room, enterprise dashboard, feedback/diagnostics, and one real file-processing download path.

Independent backend split gate:

- Do not split the backend into a separate deployable/repository before launch-test readiness unless the monorepo itself blocks deployment.
- Good timing: after `staging` and first `main` smoke tests pass, with API routes frozen and `.env`/CORS/domain contracts documented.
- First split target should be backend domain cleanup inside the existing repo: move endpoint business logic into domain services while preserving current `/api/v1/*` routes. Only after that should it become a separate backend project or independently deployed service.

## 13. Definition of Done

For documentation changes:

- The document is readable UTF-8.
- The document has no placeholder TODOs unless they are tracked as explicit backlog items.
- The Progress Ledger is updated.

For frontend changes:

- UI matches the workspace-first direction.
- Mobile and desktop layouts are checked.
- Locale text is not broken.
- Build or relevant tests pass.

For backend changes:

- Endpoint behavior remains compatible unless intentionally changed.
- Tests or targeted API checks pass.
- Security, audit, and retention impact is considered.

For cleanup changes:

- Removed paths are listed.
- Reference checks are performed.
- Generated artifacts are ignored going forward.
- No source, fixture, migration, or deployment asset is deleted without review.
